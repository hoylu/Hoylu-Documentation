Introduction & Overview

Hoylu V4

Over the past few years, Hoylu has evolved from a general-purpose whiteboard into a focused visual planning and collaboration solution for construction and project teams. That evolution has been continuous — roughly ten meaningful updates a year, each shaped by how teams were actually using the product.

Over time, a long list of improvements accumulated that couldn't be addressed through incremental changes alone — things that required a different foundation entirely. Four stood out as the clearest drivers. Large projects were hitting performance ceilings as plans grew — long-duration tasks, dense programmes, and multi-workspace setups exposed the limits of a document-based model. Project managers wanted real-time project reporting — insights that reflected the current state of a plan, not the last time someone manually synced a workspace. Organisations wanted to connect Hoylu to the rest of their toolchain — scheduling software, reporting platforms, BIM tools — without manual exports or one-off integrations. And teams wanted a mobile app their field crews could actually use on site. All four were technically impossible within the V3 architecture, where every workspace was its own self-contained file. There was no shared layer above workspaces to build on.

Key concepts

Project — The top-level container in V4. Owns the task database, shared settings (teams, labels, calendars, variance reasons), and all workspaces that belong to it.
Workspace — A planning view within a project containing a curated set of task shells. Multiple workspaces can exist within the same project, each showing a different view of the same underlying task data.
Task — A project-level record with a unique identity. Holds all task data — title, dates, duration, team, status, progress, dependencies, and history. Belongs to the project, not to any individual workspace. Tasks in V4 are live — any change is immediately reflected everywhere the task appears.
Task shell — The visual representation of a task record on a workspace. The same task can have shells in multiple workspaces simultaneously. Interacting with a shell — editing, updating status, changing dates — always writes to the underlying task record.

What changed and why

The single most important change in V4 is that task data now lives at the project level, not inside individual workspace documents. In V3, every workspace was its own island — tasks, teams, labels, and settings all lived inside a workspace file. Getting data across workspaces meant duplicating it. Getting a project-wide report meant downloading every workspace and extracting the relevant data client-side. A mobile app would have to download thousands of tasks just to show a foreman their five tasks for the week.

In V4, tasks, milestones, and constraints are stored in a central project database. A task belongs to the project — it can appear in multiple workspaces simultaneously, and any change to that task is immediately reflected everywhere, regardless of where the change came from. A status update made through a task shell on one workspace, an edit opened from a different workspace, a progress update submitted from the mobile app on site, or a date change pushed in from an external scheduling tool via the API — all of these write to the same underlying record and are visible across all views instantly. There is no sync step, no version conflict, and no workspace that holds a "fresher" version of the data than another.

A concrete foreman marks a task complete on their phone on site. At the same moment, the project manager's programme plan on their desktop reflects the update — as does the Insights dashboard and any other workspace that task appears in.

The most impactful improvements

Real-time project reporting. Project Insights now draws directly from the task database. Any change made to a task is immediately reflected in charts, PPC calculations, and the weekly work plan view. In V3, project-level insights showed the last synced state, which could be hours or days out of date. A project manager opening the dashboard on Monday morning sees the current state of the project — not Friday afternoon's sync.

A redesigned task visualization. Tasks on a plan now render as a single continuous element across their full duration — the title appears once and stays readable at any zoom level, instead of being truncated and repeated across weekly segments. The new shell adapts freely to different contexts: resize it on the whiteboard, collapse it to a small square on a floorplan, or read it in a dense overview without losing status or team information. Rendering performance is dramatically improved. A programme plan with phases spanning twelve weeks is now as fast and readable as a single-week task — and tasks stay legible even when zoomed out to see the full project.

Multiple plans per workspace. In V3, one workspace meant one plan. In V4, you can have as many plans as you need, each covering a different time range, team, or level of detail — all drawing from the same project task database. A superintendent can run a full phase plan and a 6-week look-ahead side by side in the same workspace, both reflecting the same live task data.

A structured task editor. The task editor has moved from a single scrolling panel to a multi-page design, with dedicated pages for general details, progress, connections, and history. Dependencies and task history — previously only visible on the workspace — are now accessible from the editor in any context. A foreman checking their tasks in a list view can see upstream dependencies and the full change history for a task without ever opening the plan.

Hourly planning. Plans in V4 support a new hour-level zoom, making it possible to plan and coordinate work at a finer granularity than a full day. This is particularly useful for sequencing trades on congested sites where multiple teams are working in the same area across a single day. A site manager coordinating concrete pours, formwork removal, and MEP rough-in on the same floor can lay out each activity by the hour and spot conflicts before crews arrive on site.

Project-level calendars. Each project can now have multiple calendars. Teams and individual tasks can be assigned their own calendar, so different trades with different working weeks can coexist in the same plan without interference. A concrete crew on a six-day week and a finishing trade on a five-day week can sit in the same plan with correct date calculations and off-day visualisation for each.

A public API. For the first time, Hoylu exposes a read and write API for project task data. External tools — reporting platforms, scheduling software, BIM integrations — can connect directly to live project data using service tokens, without any manual export or sync step. A Power BI dashboard can pull live PPC and milestone data from Hoylu automatically, giving executives an always-current programme view without anyone having to prepare a report.

Project-level settings. Teams, label categories, variance reasons, and calendars are now defined once at the project level and apply across all workspaces. A new Project Control Center sits alongside the workspace Control Center, with the project's own visual identity, so the scope of every setting is immediately clear. Adding a new trade team mid-project means one update in the Project Control Center — it appears in every workspace instantly, with no per-workspace configuration needed.

P6 master schedule integration. In development The XER import is being rebuilt on the V4 data model. The full XER file — activities, calendars, WBS hierarchy, activity codes, and UDFs — is imported into the project database, and planners then bring relevant subsets onto a workspace as needed. A dedicated Activity Report in Insights and write-back of actuals to P6 are also part of this workstream. See the XER Import & P6 Integration guide for full details.

What this means for your team

V4 delivers meaningful improvements across nearly every area of Hoylu, and teams should upgrade now to benefit from them. Like any ambitious product release, there are specific areas still in progress — the XER Import & P6 Integration and the mobile app are the most notable examples, and a handful of interaction models are still being finalized. These are known gaps that are actively being worked on. Hoylu has always been a product that evolves quickly in response to how teams actually use it, and V4 is no exception — it is a strong foundation that will keep getting better.

Teams who are actively running multiple workspaces on a single project will see an immediate benefit — cross-workspace reporting, shared settings, and tasks that update live across all views are available from day one. Teams managing large plans with many long-running tasks will notice the performance and readability improvements straight away. And teams with an interest in connecting Hoylu to external reporting or coordination tools now have a clear path via the API that didn't exist before. None of these require the P6 integration or the mobile app to be in place — they are valuable independently of the known gaps.

Guide 1

Central Task Database

The central task database is the architectural foundation of Hoylu V4. Rather than storing tasks, milestones, and constraints inside individual workspaces, V4 maintains them in a shared project-level database. A task belongs to the project — not to any single workspace — and can be displayed in multiple workspaces at the same time without being duplicated. What you see on a workspace is a task shell: a visual representation of a task that lives in the project database. All V4 features described in this guide require a workspace to be part of a Hoylu project — workspaces that are not part of a project continue to work in V3 mode.

The problem it solves

In V3, every workspace was its own island. Tasks existed within a workspace and only within that workspace — if you needed the same task to appear in two planning views, you had to duplicate it and keep both copies manually in sync. There was no concept of a project-level task record, which meant reporting was always scoped to individual workspaces, cross-workspace planning was effectively impossible, and there was no single source of truth for what was planned and completed across a project. A new architecture was needed — one where the task is the primary record, and workspaces are views onto it.

Key concepts

Task — A project-level record representing a unit of work. A task belongs to the project and can be brought into one or multiple workspaces as a task shell. The task holds all the planning data: title, dates, team, status, progress, labels, and connections to other tasks.
Task shell — The visual representation of a task, milestone, or constraint on a workspace. A shell is always connected to its underlying project record — any change made through a shell is a change to the record itself, and is reflected wherever else that record has a shell.
Milestone — A project-level record representing a significant point in time rather than a duration. Milestones share most task fields but have no duration.
Constraint — A project-level record used to mark an external dependency or a fixed date that cannot move. Like a milestone, a constraint has no duration — it is displayed on its required date. Constraints can be assigned to teams and have connections to other tasks like any other project element.
Workspace — A planning view within a project containing a curated set of task shells, arranged by the team for a specific planning purpose. Removing a shell from a workspace does not affect the task or its shells in other workspaces.
Project — The top-level container. It owns the task database, the shared settings (teams, labels, calendars, variance reasons), and all workspaces.

How it works

What a project contains

A Hoylu project is the container for all planning data related to this project. Every task, milestone, and constraint that has been created or imported — along with shared project settings like teams, label categories, variance reasons, and calendars are stored in one central project database. Workspaces belong to a project and draw from its task database. You can have many workspaces in a single project, each showing a different slice of the same data. Workspaces that are not part of a project are not connected to the task database and continue to operate in V3 mode.

Tasks and task shells

Every item in the task database — whether it's a task, a milestone, or a constraint — is a project-level record with a unique identity. When you place one of these items on a workspace, what you're placing is a task shell: a visual representation of that record on that workspace. The underlying task exists once in the database; the shell is the visual element to represent the task data in a workspace.

This means the same task can have shells in multiple workspaces at the same time. A task planned in a six-week look-ahead and tracked in a weekly work plan is the same record — update its status in one workspace and the change is reflected everywhere. Removing a shell from a workspace does not affect the task itself or its presence in other workspaces.

What a task contains

Each task holds a standard set of fields: title, start and end date, duration, assigned team or person, status, progress, variance reason, labels, and connections to other tasks. Milestones share most of the same fields but represent a point in time rather than a duration. Constraints are used to mark external dependencies or fixed dates that affect the plan. All three types are stored in the same database and are available across all workspaces in the project.

Adding tasks to the database

There are two ways live tasks enter the project database.

Creating tasks directly. For new projects or when adding new work, you simply open a workspace and create tasks on it. Each task created this way is immediately stored in the project database as a live task and gets a shell placed on the workspace.

Syncing an existing V3 workspace. If your project already has planning data in V3, the sync process uploads that task data to the project database — converting classic tasks into live task records. The sync does not automatically replace the classic tasks on the workspace with live task shells. After syncing, you bring live tasks onto the workspace by importing them — either individually through the import dialog or by using the "Create plan from project" tasks option. Because the V3-to-V4 migration involves decisions that vary by project, it is covered in a dedicated guide.

Converting notes to live tasks Coming soon

Whiteboard notes and text elements can be converted directly into live tasks, milestones, or constraints via the context menu — without losing their position on the workspace. The element becomes a task shell in place, and the new record is immediately registered in the project database. Bulk conversion is supported: select multiple notes and convert them all in one step. In V4, converting a note to a constraint is also available — this was not possible with classic tasks in V3.

Getting tasks from the database into a workspace

Once tasks exist in the project database, you can bring any selection of them onto a workspace at any time using the import dialog. Filter by team, label category, date range, or a combination to pull in only what's relevant for a particular planning session. Each task appears as a shell on the workspace — connected to the same underlying record as any other workspace showing that task. This makes it easy to set up focused views for specific teams or phases, run a short planning session in a temporary workspace, or build a look-ahead from a subset of the full project without duplicating any data.

Mobile app In development

The central task database architecture was built in part to support a mobile experience for field users — superintendents and trade foremen who need to view and update their tasks without opening a full desktop session. A Hoylu mobile app is in development. Because task data lives in the project database rather than inside workspace documents, the app can load only the tasks relevant to a specific user without opening in an entire workspace.

What's different from V3

Project settings — teams, label categories, variance reasons, and calendars — are now defined once at the project level and apply to all workspaces. In V3, these were configured per workspace. This reduces setup work but means changes have broader reach: updating a shared setting affects every workspace in the project at once.

  • One definition, every workspace — create a team or label category once and it is immediately available across all workspaces in the project
  • Duplicate shell vs. duplicate task — duplicating a shell creates a second view of the same underlying task; both reflect any changes made to it. Duplicating the task itself creates an independent record — changes to one do not affect the other
  • Remove shell vs. delete task — removing a shell takes a task off a workspace without affecting the task record, its presence in other workspaces, or its inclusion in project-level reporting. Deleting the task removes the underlying record from the project database and triggers a deleted state indicator across all workspaces that currently show a shell for it
  • Full change history — every update to a task is recorded automatically at the project level, regardless of which workspace the change was made from. This did not exist in V3

The full details of deletion behavior are covered in the Task Visualization guide.

What this means for your team

These are examples of how teams use the central task database in practice. Each one is independent — you can adopt any of them without the others.

Example 1 — Project overview across multiple buildings

On a multi-building project, a project manager maintains a single high-level workspace showing the main phases and milestones for each building. Each building has its own detailed phase planning workspace where the day-to-day planning happens. Because all workspaces draw from the same project task database, a milestone updated in a building-level workspace is immediately reflected in the project overview — no manual sync, no separate copy to maintain. The project workspace can also be shared directly with the owner or executive stakeholders, giving them a live, accurate view of project health at the right level of detail — without exposing them to the operational complexity of the planning workspaces below it.

Example 2 — Multiple lenses on the same plan

A main phase plan is organized by building area — floors, zones, or systems — giving the full team a spatial view of the work. Alongside it, a trade-specific workspace pulls the same tasks into a single view for one discipline, showing that trade's full workload across all areas in one place. Both workspaces show the same underlying tasks. A status update made in the trade view is immediately visible in the area plan. There is nothing to reconcile.

Example 3 — From plan to daily huddle without leaving the data

At the start of the shift, the superintendent opens a dedicated huddle workspace and imports this week's committed tasks from the project database. The tasks appear as a Kanban board — To Do, Doing, Done — giving the team a focused view without having to navigate the full phase plan. The daily huddle stays short and focused on execution. As the team marks tasks complete or flags blockers directly in the Kanban, those changes are reflected live in the phase plan. The plan stays current without anyone having to update it separately.

Frequently asked questions

Is upgrading to V4 mandatory?
Not immediately. V3 continues to work after V4 launches, and there is no forced migration. The upgrade is a deliberate step that teams take when they are ready.
Do all workspaces need to be part of a project to use V4 features?
Yes. The central task database and all V4 planning capabilities require a workspace to be part of a Hoylu project. Workspaces that are not part of a project continue to work exactly as they did in V3 and are not affected by the changes described in this guide. The goal over time is for all active workspaces to move into projects, but there is no forced migration and non-project workspaces remain fully supported.
Can I still use V3 after V4 launches?
Yes, V3 continues to work and there is no forced migration. That said, our focus going forward is V4 — it delivers meaningful improvements for projects of all sizes, not just large or complex ones. The long-term goal is to phase out V3, and while there is no set timeline for that, teams are encouraged to move to V4 when they are ready. The upgrade process is designed to be manageable, and the benefits are available from day one.
Can I use V3 and V4 at the same time, on the same project?
The two versions use different data models and do not share a live connection. You can sync existing V3 workspaces into V4 as part of a migration, but after that the V4 version of the data lives in the project database and is no longer linked back to the V3 workspace. Running both in parallel as an ongoing workflow — where changes in V3 automatically reflect in V4 or the other way around — is not supported. The recommended approach is to migrate a project to V4 when your team is ready to make the switch fully. See the V3 to V4 upgrade guide for the recommended migration process (to be written).

Task Visualization

The way tasks, milestones, and constraints appear on the Hoylu workspace has been redesigned from the ground up for V4. The goal was to make planning information clearer and more readable across all the contexts where tasks appear — not just on a plan, but on a whiteboard, in a kanban view, and when zoomed out to an overview. Tasks on a plan now render as a single continuous element across their full duration, so the title appears once and remains readable at any zoom level rather than being truncated and repeated across weekly segments. The new design adapts to available space, communicates status at a glance, and removes the rendering limitations that previously forced tradeoffs between performance and readability.

This section describes how task shells look and behave — their visual layout, how they communicate status and progress, how they adapt to different sizes and contexts, and what happens when you remove or delete them.

The problem it solves

In V3, tasks, milestones, and constraints shared a fixed visual shape regardless of where they appeared — on a plan, on a whiteboard, or in a kanban board. That shape was designed for the plan, which meant it worked reasonably well in that context but was awkward everywhere else. On the whiteboard, elements could not be freely resized, so longer titles or descriptions were often hidden behind labels or simply cut off. There was no way to expand a task to read its full content without opening the editor.

The plan view had its own structural problem. Because tasks were rendered as a separate element for each week they spanned, a single ten-week task was actually ten elements internally — glued together to appear as one. This meant the title had to be truncated and repeated on each segment, so when zoomed out only a few letters were visible per segment. It also meant that rendering a long task was proportionally more expensive: a ten-week task cost ten times as much to render as a one-week task. At project scale, this performance ceiling forced the overview mode to abandon text entirely and fall back to colored bars with no task information visible at all.

Both problems had the same root cause: the visual element and the data were inseparable. There was no room to redesign how tasks looked without also changing how they worked.

Key concepts

Task shell — The visual element placed on a workspace representing a task, milestone, or constraint from the project database. The shell is not the task — it is a view onto it. Covered in full in the Central Task Database guide.
Shell layout — The three-area structure of every task shell: the team strip on the left, the body showing title and description, and the status area at the bottom-right.
Status color bar — The thin colored line running along the bottom edge of every shell, reflecting the current task status. Remains visible even when the shell is very small or the status icon scrolls out of view.
Deleted state — The visual state a shell enters when its underlying task has been deleted from the project database. The shell remains on the workspace with a dark overlay and a trash icon, giving collaborators visibility and the option to restore or remove it. Coming soon
Remove shell — A workspace curation action that takes a shell off the current workspace without affecting the underlying task record, its reporting inclusion, or its presence in other workspaces. Coming soon
Delete task — A project-level action that flags the underlying task record as deleted across the entire project. All shells for that task enter the deleted state simultaneously. Coming soon

How it works

Visual design

Shell layout

Every task shell is divided into three areas. The left strip shows the team color and team name — this area is always visible regardless of how small the shell gets, giving a consistent visual anchor for who owns the task. The main body shows the task title and, when there is enough space, the description. The status area sits at the bottom-right and changes color to reflect the current status of the task. A thin color bar runs along the very bottom edge of the shell, ensuring the status color remains visible even when the shell is small or the status icon scrolls out of view.

Adaptive sizing

The shell adapts its layout to the space available to it. On the whiteboard it can be resized freely — shrinking to a compact square that shows only the essential team and status information, or expanding to show a full title and long description. In a kanban view it resizes vertically to reveal more content. The smallest size is suited to use cases like overlaying tasks on a floorplan, and is also prepared for future compact plan layouts including takt-style plans.

Text layout adjusts as the shell resizes. When horizontal space is limited, the title moves into the description area and scales up to remain readable. When a task has no description, the title makes use of the available space more efficiently. The shell always tries to show as much useful information as the space allows.

Multi-week tasks

In the plan view, a task that spans multiple weeks is rendered as a single continuous element. The title appears once and reads cleanly across the full width of the task at any zoom level. This replaces the segmented approach in V3 where each week was a separate element — a change that also brings a significant performance improvement, since a ten-week task now costs the same to render as a one-week task.

Title and connection dot visibility on long shells

Because task shells are now single continuous elements, a task spanning many weeks can extend well beyond the visible viewport. Two improvements address this directly.

The task title stays anchored to the visible portion of the shell. When a task starts off-screen to the left, the title remains in view rather than being hidden outside the current viewport — so you always know which task you are looking at regardless of how far the shell extends.

The connection dot tracks the cursor's horizontal position along the top edge of the shell when hovering, rather than appearing at a fixed center point. This makes it possible to initiate a connection from wherever you are on the shell, without needing to scroll to the center of a long task first.

Status colors

Each of the five task statuses has a distinct color reflected in both the status area and the bottom color bar. Status is readable at a glance even when zoomed out, without needing to read the status icon. Title and status remain visible at all zoom levels.

Team color visibility in overview mode

When zoomed out to overview level, the white overlay on the main body of the shell is reduced so the team color fills more of the shell surface. This makes it easier to read a dense plan by trade at a glance, without needing to zoom in to identify the team strip on the left edge.

Milestones and constraints

Milestones and constraints use the same shell structure as tasks but with an inverted color scheme — dark background with light text — making them easy to distinguish at a glance even in a dense plan. They each have a distinctive icon: a diamond for milestones and a stylized hourglass for constraints. Both have no duration and are displayed on a single date.

Interactions

Creating tasks

Tasks, milestones, and constraints can be created directly from the toolbar. When a workspace has live tasks enabled, the toolbar shows the V4 element types. Selecting a tool and placing it on the workspace creates a new task shell and simultaneously registers a new task record in the project database.

Status interaction

Clicking the status area of a task shell opens a status menu directly on the workspace, without needing to open the full editor. The menu shows all available status options with their respective colors, allowing a quick status change in a single step.

Replanning an incomplete task Coming soon

In V3, setting a task to Incomplete automatically prompted the creation of a replacement task linked to the original with a Start-to-Start dependency. This behaviour is returning in V4, rebuilt to take advantage of the new replanned connection type.

When a replacement task is created for an incomplete task, it will be linked using a dedicated replanned connection rather than a generic dependency. The replanned connection is excluded from all dependency validation — a task cannot be in conflict with its own replacement — and enables more precise filtering for replanned work across the project.

An incomplete task that has no replacement connected via a replanned connection will display a warning on its shell. This allows a foreman to mark a task incomplete in the field, for example on a mobile device, without needing to replan it immediately. In the next planning session the team can filter for incomplete tasks without a replacement and work through the replanning together.

The replanned connection is described in full in the Task Dependencies & Connections guide.

Removing and deleting Coming soon

🚧 The remove and delete interactions are currently in development. The design below reflects the intended behaviour — the exact implementation may evolve before release.

Because a task shell is a representation of a project-level record, removing it from a workspace and deleting it from the project are two distinct operations with very different consequences.

Removing a shell takes the task off the current workspace only. The task record remains fully active in the project database — it continues to appear in any other workspace that has a shell for it and is still included in all project-level reporting and PPC calculations. Removing a shell is a workspace curation action; it does not affect the task's existence or visibility anywhere else.

Deleting a task removes the underlying record from the project database. The task is flagged as deleted and immediately enters a deleted state in every workspace where it currently has a shell — not just the one you deleted it from. A deleted shell renders with a dark overlay and a trash icon. The task is excluded from all reporting and dashboards from the moment it is deleted.

The deleted state Coming soon

When a task is deleted, its shell remains visible on every workspace where it appeared — it does not disappear immediately. This is intentional. The deleted state serves two purposes: it gives the user who deleted the task a visible confirmation and an opportunity to restore it, and it notifies collaborators in other workspaces that the task has been removed from the project so they can react independently.

From a deleted shell, clicking the trash icon opens a small action menu with two options: Restore, which reverses the deletion and returns the task to its previous state across all workspaces, or Remove, which clears the shell from the current workspace while leaving it in the deleted state elsewhere.

Focus mode Coming soon

🚧 Focus mode support for live tasks is currently in development.

Focus mode allows you to filter the workspace to highlight only elements that match a set of criteria — tasks, milestones, and constraints that don't match are greyed out, making it easier to concentrate on a specific team's work, a status, or a label. Keyboard navigation with F and Shift+F moves between matching elements.

In V4, live task shells are included in all Focus filters alongside classic workspace elements. The available filters are: Type, Team, Assignee, Status, Label, and Label Category. A new Last Update filter is also introduced in V4, allowing you to surface tasks that have been changed within a selected time window — this filter did not exist in V3.

Where a team name matches between a project team and a workspace team, both live tasks and classic tasks appear under the same entry in the filter list.

In this first phase, navigation with F / Shift+F works only for task shells that are currently visible on the workspace. Support for navigating to non-visible elements is planned for a later phase.

Multi-instance indicator

When the same task has more than one shell on the current workspace, a small badge appears automatically on each shell showing the number of other instances.

Clicking the badge opens a panel listing the other shells of that task on this workspace, each with a navigation pin. Clicking a pin zooms the workspace to that shell.

Tab / Shift+Tab cycles between shells of the same task. This works for shells currently visible on the workspace.

Make unique — available from the context menu when a shell has other instances on the same workspace — creates a new independent live task from that shell. The shell is disconnected from the shared task record and becomes its own standalone task in the project database. The badge on any remaining shared shells updates immediately to reflect the reduced count.

Duplication shortcuts: Ctrl+D creates a full Nexus duplicate — a new independent task with a new shell. Ctrl+Shift+D creates a shell-only clone — a new shell pointing to the same underlying task record, which is what triggers the multi-instance indicator.

What's different from V3

The visual element on the workspace was the task in V3 — the representation and the data were inseparable. The V4 task shell is a purpose-built visual component that is independent of the data it represents, which makes a number of improvements possible.

  • Single element per task — in V3, a task spanning multiple weeks was rendered as a separate element per week, glued together internally. In V4 it is a single continuous element. The title appears once across the full width and is readable at any zoom level

Task Editor

The task editor is the primary interface for viewing and editing the full details of a task, milestone, or constraint. It is organised into a set of focused pages — each covering a specific aspect of the task — and is designed to work consistently across the desktop application and the mobile app.

The problem it solves

In V3, task information was split across two places. Some details — title, dates, team, status — were accessible through the editor panel. But other information, like dependencies or labels, was only visible on the task visualization directly on the workspace. This meant that any context where tasks were shown differently — a list view, a filtered view, or eventually a mobile screen — could only ever show a subset of the task's full picture. There was no single place where all task information was consistently accessible regardless of how or where you were looking at the task.

The new task editor was built to be that single place. By organising all task information into a structured set of pages, the editor can surface the full picture — details, progress, connections, history — in any context it appears in. It is also built as a shared component that can show more or fewer pages depending on the context, which means the same editor works across the desktop application and the mobile app without requiring separate implementations.

Key concepts

Task editor — The panel that opens to display and edit the full details of a task, milestone, or constraint. Organised into pages, each covering a specific category of information.
Editor page — A focused view within the task editor dedicated to one aspect of the task. The pages available depend on the context the editor is opened from and the settings of the workspace.
Status — The current state of a task in the planning cycle. Reflected visually by both a label and a color throughout the editor and on the task shell. The five states are: Unplanned, Planned, Committed, Complete, and Incomplete.
Progress — A percentage value indicating how much of the work on a task has been completed. Distinct from status — a task can be in progress without its status having changed.
Variance reason — The recorded explanation for why a committed task was not completed as planned. Selected from a project-level list defined in the Project Control Center.
Outcome notes — A free-text field for adding context to a variance reason — what specifically happened, what the impact was, or what the next step is.
Dependency — A relationship between two tasks that defines the order in which work should happen. Managed on the Connections page and covered in detail in the Task Dependencies & Connections guide.
History entry — A recorded snapshot of a change made to a task, including what changed, who made the change, and when. Entries are created automatically whenever a task is updated.

How it works

Navigation

The editor is organised into pages. You move between them using the dropdown at the top of the editor, or by clicking the page indicator dots at the bottom. The number of dots reflects how many pages are currently available — some pages can be enabled or disabled at the workspace level, so the editor may show more or fewer pages depending on how the workspace is configured.

General page

The General page is the main details view for a task. It shows the task title, assigned team or person, start date, due date, duration, and current status. The status button reflects the current state through both label and color, and can be changed directly from this page. Optional time fields allow start time and due time to be set when hour-level precision is needed. Variance reason and outcome notes are also managed here — available when a task has not been completed as planned.

Progress Update page

The Progress Update page shows the current completion percentage for a task and allows it to be updated directly. This page can be enabled or disabled at the workspace level — it is typically active in workspaces where teams are tracking weekly progress, and hidden in workspaces where that level of detail is not needed. The intent for this page is to evolve into a running daily log of progress updates over the life of a task, giving a more detailed view of how work advances between planning cycles. That fuller log capability is planned for a future release.

Connections page

The Connections page lists all dependencies and links associated with the task. In V3, dependency information was only visible on the task visualization on the workspace — there was no way to see or manage connections from the editor itself. In V4, the Connections page brings this information into the editor, making it accessible regardless of how or where the task is being viewed. Dependencies show connected tasks and whether they are upstream or downstream. Links are external references such as URLs to documents or resources. From this page you can navigate directly to a connected task or remove a connection. Dependency conflict warnings — flagging cases where a task's schedule is in tension with connected tasks — are coming soon to this page.

History page

The History page is a read-only log of every recorded change to the task. Each entry shows when the change was made, who made it, and what changed — including status, dates, and crew size. Fields that changed since the previous entry are highlighted, making it easy to see what shifted between planning cycles. The history captures changes regardless of which workspace or context the edit was made from.

A dedicated Labels page is the next planned addition to the editor. Labels can already be assigned to tasks today through the existing label assignment dialog on the workspace — the new page will bring that management directly into the task editor.

The editor has moved from a single scrolling panel to a structured multi-page design, and several capabilities that were either missing or only accessible on the workspace are now available directly within the editor.

  • Multi-page structure — task information is organised into focused pages rather than a single form, making it faster to navigate and easier to extend with new capabilities over time
  • Time fields — start time and due time can now be set for hour-level precision, which was not available in V3
  • Calendar picker — a task can now be assigned a specific project calendar, ensuring date calculations respect the correct working days
  • Connections page — dependencies and links are now viewable and manageable directly in the editor. In V3, connection information was only visible on the task visualization on the workspace, making it inaccessible in list views or other contexts
  • History page — a full, read-only audit trail of every change made to a task, recorded automatically. This had no equivalent in V3
  • Extensible by design — the page structure is built to accommodate new pages over time. Labels are the next planned addition

During a weekly look-ahead session, a superintendent notices that a concrete task has shifted two weeks later than originally planned. Rather than asking around, they open the task editor and navigate to the History page. Task Dependencies & Connections

Connections define relationships between tasks, milestones, and constraints. In V4, project-level connections are stored in the central task database — they belong to the project, not to a specific workspace, and are visible wherever a task has a shell. There are two types of connection in a V4 project workspace: dependencies between project elements, and links to external resources such as URLs. Workspace-only elements like notes can still be connected with visual lines on the whiteboard, but these are not stored in the project database.

The problem it solves

In V3, connections grew organically from the whiteboard model. The original connection type was a visual line between any two elements — useful for showing relationships but carrying no structured meaning. A dependency type was added later, introducing typed relationships (FS, SS, FF, SF), lag values, and order warnings. Both types coexisted, and both could be freely restyled — color, line width, and line style were all configurable per connection.

This flexibility had a cost. Because connection styling was freeform, there was no consistent visual language for what a connection meant. Color and style choices were personal and inconsistent across teams. More importantly, as V4 moves task data to the project level, connections between tasks need to be stored and validated at the project level too — not as workspace-level drawing elements. A connection drawn between two shells in one workspace needs to be the same connection visible in any other workspace where those tasks appear.

V4 addresses this by storing dependency connections in the project database and moving toward a semantic visual language where color and style always communicate something specific about the connection, rather than being decorative choices.

Key concepts

Dependency — A typed relationship between two project elements defining the order in which work should happen. Stored in the project database and visible across all workspaces. Types: FS, SS, FF, SF.
Lag — A duration offset applied to a dependency. A positive lag adds a gap between the predecessor and successor; a negative lag (lead) allows overlap.
Upstream / Downstream — The direction of a dependency relationship. The predecessor is upstream; the successor is downstream.
Dependency warning — A flag raised when a task's dates conflict with its dependency relationships. Validated server-side and shown in red on the connection and on the Connections page in the task editor.
Link — An external reference attached to a task, such as a URL to a document or resource. Managed in the task editor and visible across all workspaces.
Workspace connection — A visual line between non-project elements on the whiteboard. Stored at the workspace level only, retains full styling options, and is not synced to the project database.

How it works

Drawing connections

Connections between task shells can be drawn directly on the whiteboard — the same interaction as V3. Hovering over a task shell reveals connection points, and dragging from a connection point to another shell creates a dependency. The connection is immediately stored in the project database and appears in the Connections page of the task editor for both tasks involved.

Dependency types

V4 supports four dependency types, following standard CPM scheduling conventions:

  • Finish to Start (FS) — the successor cannot start until the predecessor finishes. The most common dependency type
  • Start to Start (SS) — the successor cannot start until the predecessor starts
  • Finish to Finish (FF) — the successor cannot finish until the predecessor finishes
  • Start to Finish (SF) — the successor cannot finish until the predecessor starts. Rarely used

Dependencies can also include a lag value — a positive or negative duration that offsets the relationship. A positive lag adds a gap between the two tasks; a negative lag (lead) allows overlap.

Visual language

In V4, the visual appearance of a connection communicates its state automatically — there are no manual styling options for project-level dependencies. Connections that are causing a schedule warning are shown in red, making it immediately clear which relationships are in conflict without having to inspect each task individually.

The goal is for all connection styles to carry semantic meaning — visual properties will be added over time to represent new relationship types as they are introduced.

Dependency warnings

Dependency warnings are validated server-side, which means they apply regardless of which workspace a task is viewed from or whether all connected tasks are currently visible on the whiteboard. If a task's dates conflict with its dependency relationships, a warning is shown on the connection and flagged on the Connections page in the task editor. This server-side approach ensures warnings are consistent across all workspaces and accessible to the mobile app without requiring all tasks to be loaded.

Links

In addition to dependencies, a task can have links — external references such as URLs to documents, drawings, or other resources. Links are managed on the Connections page in the task editor and are visible in all workspaces where the task has a shell.

Workspace-only connections

Visual line connections between non-project elements — notes, shapes, and other whiteboard content — continue to work as before. These connections retain their full styling options (color, width, style) and are stored at the workspace level. They are not synced to the project database and do not appear in the task editor or project reporting.

Assistive rescheduling

When a dependency conflict exists between two tasks, a warning border appears on the affected task and a red triangle appears in the connection menu. Clicking the triangle opens a resolution dialog with four options: run the tasks in parallel (converting to a Start-to-Start dependency), push the conflicting task to a later date, pull it to an earlier date, or adjust the lag. Push can also be triggered with Ctrl+A and pull with Ctrl+Shift+A. Both tasks need to be visible on the same workspace for the conflict indicator and resolution options to appear.

Replanned connection Coming soon

When a task is marked Incomplete, Hoylu will prompt the creation of a replacement task. The original and its replacement are linked with a dedicated replanned connection — a distinct connection type that differs from standard dependencies in several important ways.

The replanned connection is excluded from all dependency validation. A task cannot be in a scheduling conflict with its own replacement, so replanned pairs are never flagged with warnings regardless of how their dates relate to each other.

Replanned connections also enable more precise filtering. Teams can filter for incomplete tasks that have no replacement yet connected — useful in planning sessions where foremen may have marked tasks incomplete in the field without replanning them on the spot. The plan can surface these as a distinct group, so the team can work through them together.

An incomplete task without a replanned replacement will display a warning on its shell, making unresolved replanning visible at a glance without disrupting the ability to mark tasks incomplete quickly in the field.

The replanned connection is visually distinct from standard dependency connections, and like all project-level connections it is stored in the project database and visible across all workspaces where the connected tasks have shells.

Creating linked tasks Coming soon

V4 introduces a set of shortcuts for quickly creating a new live task in a predefined relationship to an existing one — a predecessor, a subtask, or a successor. All tasks and connections created this way are stored in the project database as live records immediately.

Predecessor — a new task placed one working day before the selected task's start, linked with an FS dependency (predecessor → selected task).

Subtask — a new task placed at the selected task's start date and bounded by it, linked with SS and FF dependencies to keep it within the parent's duration.

Successor — a new task placed on the first working day after the selected task ends, linked with an FS dependency (selected task → successor).

Each newly created task inherits the calendar of the task it was created from, so working day placement is always consistent.

These can be triggered from the context menu (quick single-task create), the connection dot radial menu, or a multi-subtask modal that splits a task's duration into several subtasks at once — by total length, by number of working days, or by day of week. A fourth entry point allows inserting a new task inline into an existing connection, replacing it with two new connections to the inserted task.

What's different from V3

  • Project dependencies are stored in the database, not on the whiteboard — in V3, connections were workspace-level drawing elements. In V4, dependencies between tasks are stored in the project database and appear in every workspace where the connected tasks have shells
  • No manual styling for project connections — in V3, all connections could be recolored and restyled freely. In V4, project dependency appearance is set automatically based on connection state and type. Manual styling is retained only for workspace-level line connections between non-project elements
  • Semantic visual language — connection color and style now communicate meaning. Regular dependencies are grey, warning states show in red. Additional semantic styles will be added as new connection types are introduced
  • Server-side dependency validation — warnings are validated at the project level rather than client-side, ensuring they are consistent across all workspaces and accessible regardless of which tasks are loaded on the whiteboard
  • Replanned connection coming — a dedicated connection type will link incomplete tasks to their replacements, excluded from dependency validation and enabling filtering for unresolved replanning Coming soon

What this means for your team

Example 1 — A conflict visible across workspaces

A trade foreman moves a task in the phase planning workspace — pushing it a week later to reflect a material delay. That task has a Finish-to-Start dependency with a project milestone. The PM, working from a separate high-level programme workspace, sees a red warning outline appear on the milestone shell — indicating that something upstream is now in conflict with it. They don't need to be in the phase planning workspace to know something needs attention, and the trade task doesn't need to be imported into the programme workspace. The validation happens server-side against the full project data, surfacing the conflict wherever the milestone has a shell.

Example 2 — Dependency warnings on the whiteboard during a pull planning session

During a pull planning session, the team is working on the whiteboard — tasks placed freely, not yet arranged in a formal plan. Some tasks already have dates assigned from earlier discussions. As the facilitator draws a dependency between two tasks to capture a handoff agreement, a warning appears immediately on the connection — the successor's start date is earlier than the predecessor's finish date. The conflict is visible right there in the session, before it ever makes it into the plan, giving the team the chance to resolve it in the room.

Plan Visualization

The plan is the primary structured layout for scheduling and tracking work in Hoylu. It organises task shells into a timeline with swimlanes — a structure familiar from V3. In V4 the plan is built on a new technical foundation that improves performance, adds new timeline zoom levels, and lays the groundwork for tighter integration between how tasks are arranged in a plan and how they are reported on across the project.

The problem it solves

In V3, each workspace could contain only one plan, and that plan was fixed in place — it could not be moved or repositioned on the workspace. All tasks shared a single workspace-wide calendar, swimlanes could not easily be reordered, and the timeline supported only week and day zoom levels. These were practical limitations, but the deeper structural issue ran further.

In V3, tasks didn't actually store their own dates. The plan had a fixed reference point — an anchor date — and a task's dates were calculated on the fly based on where it sat on the canvas relative to that anchor. This worked well enough when Hoylu was primarily a visual planning tool, but created significant problems as the product became more data-driven. Changing the plan's anchor date did not move any tasks — it silently recalculated all their dates, effectively replanning the entire workspace in one step. Adding a public holiday after tasks had already been placed could shift end dates even on tasks that had already been completed. The plan was the source of truth for task dates, not the task itself.

The same principle applied to swimlanes. In V3, a task's swimlane position was used as a proxy for reporting — which trade or area a task belonged to was determined by where it sat on the workspace, not by any data field on the task. This meant that moving a task to a different swimlane changed how it was reported on, and tasks viewed in a list or any context outside the plan had no reliable way to communicate that grouping information.

Performance was also a growing constraint. As covered in the Task Visualization guide, tasks were rendered as one element per week, which meant long tasks were disproportionately expensive to render and limited how many tasks a plan could show at reasonable performance.

Cleanup — the user-triggered operation that sorts and repositions task shells within a swimlane — also carried a fundamental limitation in V3. The V3 algorithm was greedy: it aimed for the smallest possible swimlane height by filling every available slot as tightly as it could. This kept the plan compact, but at a significant cost to readability. There was no apparent structure to where tasks ended up, adding or removing a single task could reshuffle the entire arrangement, and the dense packing made it difficult to follow dependency connections between tasks. The V4 cleanup replaces this with the Structured Waterfall — deliberately trading some space efficiency for a layout that is far easier to read at a glance. An open limitation remains: users who manually arrange a swimlane to their preference have no way to protect that layout from being overwritten if cleanup is triggered. We are aware of this and are considering options, including a per-swimlane opt-out.

Key concepts

Plan — A timeline element placed on a workspace, containing one or more swimlanes. Connected to the project's task database. Multiple plans can exist in the same workspace.
Swimlane — A horizontal row within a plan representing a grouping of tasks. In V4, swimlanes will be linked to a data dimension — team, label category, or WBS code — so that a task's swimlane position reflects a real field on the task record. Coming soon
Timeline — The date header running across the top of a plan. Controls the visible date range and zoom level. Clickable to change zoom level for any segment.
Zoom level — The granularity at which the timeline displays time. Available levels: week, day, and hour. Collapsed week is coming soon.
Overflow column — A column at either end of the plan's active date range showing tasks that fall outside it. Prevents tasks from being hidden when the plan's date range is adjusted.
Anchor date — (V3 term, for reference) In V3, the reference date used to calculate task dates from their position on the whiteboard. This concept no longer exists in V4 — task dates are stored on the task itself.
Cleanup — A user-triggered operation that sorts and repositions task shells within a plan. Accessible from the plan context menu, which applies it to all swimlanes simultaneously, or from an individual swimlane context menu, which applies it to that swimlane only. In V4, cleanup uses the Structured Waterfall algorithm.
Structured Waterfall — The cleanup algorithm introduced in V4. Divides the swimlane timeline into configurable segments and arranges tasks in a cascading sequence within each segment, ordered by start date and then duration. Produces a chronologically coherent layout that prioritises readability over space efficiency. The segment interval defaults to 4 weeks and is configurable at plan and swimlane level.

How it works

The plan and swimlanes

A plan is a timeline element placed on a workspace. It contains one or more swimlanes, each representing a row of tasks arranged along the timeline. Multiple plans can be placed on the same workspace — each showing a different slice of the project's task database, covering a different time range, a different set of teams, or a different level of detail. Plans can be freely repositioned on the whiteboard and swimlanes can be reordered within a plan by dragging.

The timeline

The timeline header runs across the top of the plan and defines the time range displayed. Clicking a segment on the timeline opens a menu that lets you change the zoom level for that portion of the plan. Four zoom levels are available:

  • Collapsed week Coming soon — the most compact view, showing multiple weeks in a narrow column. Suited to long-running projects or overview planning such as multi-year infrastructure projects
  • Week — the standard view, showing one column per week
  • Day — expands a week into individual days, useful for short-duration work and daily planning
  • Hour — expands a single day into hours, available for work that requires hour-level precision

Jumping from week directly to hour is not available — you must expand to day first. The current date is marked with a line across the plan, and the current week or day is highlighted in the timeline header.

Overflow columns

Tasks that fall outside the active date range of a plan are not hidden — they appear in overflow columns at either end of the timeline. This means you can adjust a plan's date range without risk of tasks disappearing. Overflow columns give a clear visual signal that tasks exist outside the current view, and the plan can be extended to bring them back into the main timeline at any time.

Swimlanes and data Coming soon

🚧 The swimlane-data coupling described below is actively in development and not yet fully shipped. We are being explicit about the direction here because it affects how reporting works and how users should think about structuring their plans going forward.

In V3, a task's swimlane position was used as a proxy for filtering in reporting — which swimlane a task sat in determined how it appeared in insights, the weekly work plan view, and the constraint log. In V4 this changes fundamentally: the swimlane position is replaced by actual data fields on the task.

In V4, each plan will be linked to a data dimension — team, label category, or in future WBS code. A plan linked to "Team" shows one swimlane per team. A plan linked to an "Area" label category shows one swimlane per area value. The swimlane a task shell appears in reflects a real data field on the task — not just its position on the whiteboard.

This means that if you want a task to appear in a specific swimlane, the corresponding label on the task needs to be set. At the start, users may need to assign labels manually to tasks before they appear in the correct swimlane. In future, dropping a task shell into a swimlane will automatically update the corresponding data field on the task.

If a data field is updated elsewhere — in the editor, through the API, or in another workspace — and the task's label no longer matches the swimlane it sits in, a warning will appear on the task shell. The user can then resolve the conflict by either moving the shell to the correct swimlane or updating the data to match the current position. This is the same principle as the deleted state: the plan shows you that something has changed and gives you the agency to decide what to do about it.

The benefit of this approach is that reporting is no longer tied to whiteboard position. The same task can appear in multiple plans — each slicing the data in a different way — without any ambiguity about which team it belongs to or which area it covers. That information lives on the task itself and is consistent across every workspace and every view.

This is a meaningful shift for teams coming from V3. In V3, the plan was effectively a data store — the swimlane a task sat in was how the system knew which trade or area it belonged to. Because the plan held that information, deleting it meant losing it, which is part of why V3 plans became difficult to remove or restructure once work was underway. In V4, that information lives on the task itself as a label. The plan is a view onto it — not the source of it. A plan can be created, adjusted, or removed without affecting the underlying task data. This opens up workflows that weren't previously practical: running the same data through different plan layouts, creating a focused plan for a specific session, or building a new plan for an incoming workspace without having to restructure existing data.

Plan presets Coming soon

To make the link between label categories and plans tangible, V4 will offer plan presets in the asset panel — one for each project team and one for each label category. Dragging a preset onto the workspace creates a ready-made plan with swimlanes pre-configured from that category's labels, starting from the current week with a 4-week default range. Swimlane colors follow the label colors; swimlane order follows the label order in the category. This makes it possible to spin up a structured, data-linked plan in a single drag rather than building it swimlane by swimlane.

Off-days

Off-days are no longer visualized at the plan level. In V4 each task shell is responsible for showing its own non-working days, since a single plan can now contain tasks governed by different project calendars. Tasks that span non-working days show those days with a hatched overlay directly on the shell.

How calendars are created, assigned to teams and tasks, and how the hierarchy resolves when multiple calendars apply is covered in the Calendars guide.

Cleanup & resize

Cleanup is a user-triggered operation that sorts and repositions all task shells within a swimlane. It is available from both the plan context menu and the individual swimlane context menu. Running it from the plan applies the Structured Waterfall to all swimlanes simultaneously — useful after a bulk import or a major replanning session. Running it from a swimlane applies it to that swimlane only, leaving the rest of the plan untouched.

The V4 cleanup uses the Structured Waterfall algorithm. Rather than packing tasks as tightly as possible, it divides the swimlane timeline into segments and arranges tasks in a cascading sequence within each segment. Tasks are ordered by start date, with duration as a tiebreaker for tasks starting in the same period. The result is a layout with an immediately apparent chronological structure — a planner scanning the board can orient themselves by segment at a glance, and the cascading arrangement within each segment makes sequencing and dependency connections between tasks easy to follow.

The segment interval defaults to 4 weeks but is configurable at the plan level, with the option to override on individual swimlanes. This allows different swimlanes within the same plan to use intervals suited to their content — a milestone swimlane might use a 1 or 2 week interval, a phase swimlane a 6 month interval, and the remaining swimlanes the standard 4 week default.

When tasks overlap — which can occur with the Structured Waterfall approach — later-starting tasks are always rendered in front of earlier ones, keeping the most recent work visible.

The Structured Waterfall is a new approach and we are actively gathering feedback on how it works across different project types and planning styles. If it is not producing the layout you need, we would welcome your input — it will directly shape how cleanup develops.

What's different from V3

The plan looks familiar but a significant number of things work differently underneath.

  • Dates drive position, not the other way around — in V3, a task's dates were inferred from its position relative to the plan's anchor date. In V4, task dates are stored on the task itself and the plan renders tasks where their dates say they should be. Adjusting the plan's date range automatically repositions all shells to reflect the new view
  • Multiple plans per workspace — V3 allowed only one plan per workspace. V4 has no limit
  • Plans can be freely repositioned — plans can be moved anywhere on the whiteboard by dragging. In V3 the plan was fixed in place
  • Swimlanes can be reordered — swimlanes can be dragged into a different order within a plan
  • Four zoom levels — week, day, hour, and coming soon: collapsed week. V3 supported only week and day
  • Overflow columns — tasks outside the active date range are shown in overflow columns rather than disappearing
  • Swimlanes linked to data, not whiteboard position — in V3, swimlane position was used as a proxy for reporting. In V4, swimlanes will be linked to data fields on the task, making reporting independent of whiteboard position Coming soon
  • Off-days on the task, not the plan — non-working days are shown on the task shell itself, allowing tasks with different calendars to coexist in the same plan
  • Significantly better performance — the new plan component reduces render cycles and handles large numbers of tasks substantially more efficiently than V3
  • Structured Waterfall cleanup — the V3 compact cleanup aimed for the smallest possible swimlane height, filling gaps aggressively at the cost of readability and stable task positioning. V4 replaces this with the Structured Waterfall: tasks are arranged in cascading segments with a configurable interval, defaulting to 4 weeks, adjustable per swimlane. The result is a more readable, chronologically coherent layout that stays stable as tasks are added or removed

Project Insights

Project Insights is a reporting panel that gives teams a live view of planning data across a project. It draws directly from the central task database, which means any change made to a task — from any workspace, the mobile app, or the API — is reflected in the charts and views immediately, without any manual sync step.

The problem it solves

Reporting in V3 was split across two separate components with different capabilities. Workspace insights offered charts, a task list, a weekly work plan view, and a constraint log — but only for the tasks within that specific workspace. Project-level insights existed but were more limited, offering only charts and a task list, and crucially they were never live. Project insights showed the last synced state of the data — meaning a project manager viewing the charts could be looking at information that was hours or days out of date, depending on when someone had last triggered a sync. Getting a current picture of the project required opening individual workspaces and checking insights there.

The sync mechanism itself was the same batch operation now used to migrate V3 workspaces to V4 — it read workspace documents and pushed their task data into a separate reporting database. This was always a workaround for the underlying architecture: because task data lived inside workspace documents, there was no way to report on it in real time without first extracting it.

The charts and task list were also combined into a single view, where you had to choose which chart to display and configure the date range to match. Switching between chart types meant updating filters manually to make the numbers meaningful. Filters ran across the top of the panel, leaving limited space — not enough room for label category filtering.

In V4, that workaround is no longer needed. Task data lives in the project database from the moment a task is created or updated. Insights draws directly from that database, so the data is always current.

Key concepts

PPC (Percent Plan Complete) — The percentage of tasks that were committed for a given week and completed as planned. The primary metric for measuring plan reliability in the Last Planner System methodology.
KPI section — The summary area at the top of the chart view, showing headline tiles and three overview charts: Task PPC, Task Status distribution, and Variance reason distribution.
Weekly Work Plan (WWP) — A view showing tasks for a selected period with their completion status. The source data for PPC calculations.
Constraint Log — A view listing all constraints in the project, showing which are resolved, outstanding, or past due.
Live data — In V4, Insights reflects the current state of all tasks in real time. There is no sync step required. Any change to a task is immediately visible in all views.

How it works

One component, two contexts

Insights is a single shared component that appears in two contexts. At the project level it gives a view across all workspaces and all tasks in the project, with the option to filter down to a specific workspace. Inside a workspace, the same component appears with the workspace filter pre-applied — you see only the tasks relevant to that workspace, and the filter cannot be removed. The views and charts are identical in both contexts; only the scope changes.

Filters

Filters appear in a panel on the left side of the Insights panel. You can filter by workspace, team, assignee, status, and label category. The filter panel applies to all four views — charts, task list, weekly work plan, and constraint log — and filters are preserved when switching between views, so you don't need to reconfigure them each time.

Chart view

The chart view is organised into two areas. At the top, a KPI section shows summary tiles and three overview charts side by side: Task PPC, Task Status distribution, and Variance reason distribution. The summary tiles show headline numbers including total tasks, milestones, constraints, PPC, uncommitted tasks, and tasks past due. These give a quick health check of the project at a glance.

Below the KPI section, a scrollable area shows additional charts at full width. The PPC trend chart shows how plan reliability has changed over time — it can be displayed as a line chart showing PPC by week, or as a bar chart breaking down PPC by team, with both views available side by side. Each chart has its own independent date range, so the PPC trend can look back over completed weeks while the status chart looks forward over upcoming work — without requiring you to change a shared filter that would affect all views at once.

Task list

The task list shows all tasks matching the current filters in a tabular format. Columns include type, title, week, crew size, swimlane, status, label, and assignee. PDF export is available. CSV export is coming at a later stage.

Weekly Work Plan view

The Weekly Work Plan view shows tasks for a selected period with their completion status. It is the primary view for tracking plan reliability — the data that drives PPC. At project level this view is new in V4; it was previously only available within individual workspaces.

The grouping options have been significantly expanded. In V3 you could group by team or by swimlane. In V4 you can group by team, by status, or by any label category — for example grouping by an "Area" label category to see work broken down by zone or location. Grouping by label category replaces grouping by swimlane, reflecting the broader shift away from whiteboard position as a data proxy described in the Plan Visualization guide.

The view also supports sorting by multiple columns — start date, due date, status, team, or title. In V3 the WWP always sorted by start date with no alternative.

Constraint Log

The Constraint Log lists all constraints in the project — external dependencies and fixed dates that affect the plan. It shows which constraints are resolved, which are outstanding, and which are past due. At project level this view is new in V4; it was previously only available within individual workspaces.

Updated PPC calculation

The way PPC is calculated has changed in V4. In V3, a task that spanned multiple weeks contributed to the PPC score for each of those weeks — completing a five-week task improved PPC across five separate week buckets. In V4, a task contributes to PPC only in the week it is completed. This gives a more accurate picture of week-by-week plan reliability. Additionally, future tasks are excluded from PPC calculations — if a selected date range extends into the future, only completed weeks contribute to the score. A date range set entirely in the future will show PPC as not applicable.

Bulk Import & Workspace Sync

The goal is for all workspaces to move to V4. For existing customers with V3 workspaces, Hoylu provides a migration path to upload workspace data to the project database and bring it back as live task shells. Sync is a one-time operation — once a workspace is fully migrated, it is no longer needed.

For teams starting fresh, skip the sync entirely — create tasks directly on a V4 workspace and they go straight into the project database from day one.

For existing customers, the migration flow is:

  1. Upload V3 workspace data to the project database via the sync wizard
  2. Re-import the upgraded tasks onto the workspace as live task shells
  3. Remove the original V3 task data from the workspace to avoid running both in parallel

This section covers the sync wizard in detail, along with the import and create plan tools used to populate workspaces from the project database.

The problem it solves

As soon as a project has more than one workspace, data consistency becomes a challenge. Teams, labels, and variance reasons are defined per workspace in V3 — which means the "Electrician" team in workspace 1 and the "Electrician" team in workspace 2 may look the same but are entirely separate records. When task data from both workspaces is uploaded to the project database, the system needs to know whether these are the same team or two different ones. If they are the same, tasks from both workspaces should roll up into the same team in reporting. If they are different, they should stay separate.

The same problem applies to labels and variance reasons. Without a way to resolve these conflicts explicitly, cross-workspace reporting would silently produce misleading results — showing the same work under multiple teams or labels rather than aggregating it correctly.

The sync wizard makes this explicit. Before task data is uploaded, shared elements are compared and any conflicts are surfaced for the user to resolve. Only after that decision is made does the task data upload proceed — ensuring the project database has a clean, consistent foundation for reporting.

The import and create plan operations address the inverse problem: once data is in the project database, planners need a way to pull relevant subsets of it back onto a workspace without loading everything at once.

Key concepts

Sync — The process of uploading workspace-level data (teams, labels, variance reasons, tasks, and connections) to the project database. Runs in three phases with explicit conflict resolution for shared elements. During sync, several automatic operations are also applied: V3 plan swimlanes are converted to label categories to support plan recreation in V4, lines between tasks are converted to FS dependencies, and task titles over 100 characters are split into title and description.
Conflict — A mismatch between a workspace record and a project record with the same name but different properties — for example, a team with a different color, or a label in a different category. Resolved by choosing which version to keep, or keeping both with a unique name for one of them.
Import — Selecting tasks from the project database and placing their shells on the current workspace, filtered by team, label, date range, or other criteria.
Create plan from project tasks — A one-step operation that generates a new plan with task shells placed automatically based on a chosen date range and swimlane structure.

How it works

Sync local data with project

The sync wizard runs in three phases, each requiring confirmation before proceeding.

Phase 1 — Shared elements. Teams, label categories, labels, and variance reasons are compared between the workspace and the project database. Where there is a clean match, the data is merged automatically. Where a conflict is detected — for example, a team with the same name but a different color, or a label assigned to a different category — a conflict resolution screen is shown. For each conflict, you can choose to keep the workspace version, keep the project version, or keep both (in which case you must provide a unique name for one of them). Calendar data is also converted and uploaded in this phase.

Phase 2 — Task data. Task records are uploaded to the project database with their dates, status, team assignments, labels, and other fields. Each task is registered as a project-level record. The sync does not automatically replace the classic tasks on the workspace with live task shells — after syncing, you bring live tasks onto the workspace by importing them or using the Create plan from project tasks option.

Phase 3 — Task relations. Dependency connections between tasks are uploaded as project-level connection records.

Conflict resolution can appear on any sync run — not just the first. If data has drifted between workspace and project since a previous sync, new conflicts will be shown.

The sync menu item may be hidden automatically when there is nothing left to sync. (Feasibility to be confirmed)

What sync does automatically

In addition to uploading tasks, connections, and shared elements, the sync performs several automatic operations to make the transition to V4 smoother.

Swimlane label categories. For each workspace that contains a V3 plan, sync creates a new label category named "Swimlanes for WS [workspace number]" and a label for each swimlane in that plan — preserving the swimlane color and order. This gives you everything you need to recreate the same plan layout in V4. Since V3 allowed only one plan per workspace, there will be at most one swimlane label category per workspace.

To recreate the plan in V4, use either Create plan from project tasks Coming soon or the Import dialog, and choose the option to organise swimlanes by label category — then select "Swimlanes for WS [workspace number]". The resulting plan will have the same swimlane structure as the original V3 plan, with tasks placed at their correct dates.

Additional fixes applied during sync:

  • Lines drawn between tasks in V3 are converted to FS (Finish to Start) dependencies and uploaded as project-level connections
  • Task titles longer than 100 characters are automatically split — the first 100 characters are kept as the title and the remainder is moved to the task description
  • Broken label assignments on older workspaces are repaired during upload

Import live elements from project

The import dialog lets you bring task shells from the project database onto the current workspace. You can filter by team, label category, status, and date range to narrow down which tasks to import. Selected tasks are placed on the workspace as shells — not copies, but references to the same underlying project records. Milestones and constraints can be imported alongside tasks.

This is the day-to-day tool for curating workspace content — adding tasks for a specific team, pulling in an upcoming date range, or creating a focused view for a particular planning session.

Create plan from project tasks

The create plan option generates a new plan on the workspace in one step. You choose an optional date range and a swimlane structure — for example by label category or by team — and the plan is created with task shells placed automatically based on their dates and the grouping you selected. This is a faster alternative to importing tasks manually and then creating and configuring a plan separately.

What's different from V3

  • Workspace data can now be uploaded to the project — in V3, teams, labels, and variance reasons were workspace-local and could not be shared or aligned across workspaces. The sync wizard promotes these to the project level with explicit conflict resolution
  • Cross-workspace consistency is now explicit — when the same team or label exists in multiple workspaces under slightly different names or colors, the conflict is surfaced and resolved before task data is uploaded. This ensures reporting is accurate rather than silently duplicating records
  • Selective import replaces manual duplication — in V3, getting tasks from one workspace into another required duplicating them manually. The import dialog lets you pull any subset of project tasks into a workspace as shells
  • Create plan from project tasks is new — there was no equivalent in V3. A complete plan with task shells can be generated automatically from existing project data in one step
  • V3 plan swimlanes are migrated as label categories — since V4 swimlanes are linked to data fields rather than whiteboard position, the sync automatically converts each V3 swimlane into a label and groups them into a label category per workspace. This gives you a direct starting point for recreating the same plan layout in V4. See the Plan Visualization guide for how swimlanes and data dimensions work in V4

What this means for your team

Example 1 — Setting up a focused 6-week look-ahead from existing project data

A superintendent has just finished migrating three V3 workspaces to V4. All task data is now in the project database. Rather than manually recreating a look-ahead workspace, they use "Create plan from project tasks" — selecting a 6-week date range and choosing to organise by team. The plan is generated in one step with swimlanes for each trade and all relevant tasks placed automatically at their correct dates. What would have taken twenty minutes of manual importing and configuring takes about thirty seconds.

Example 2 — A temporary workspace for a focused planning session

A lean lead is running a pull planning session for a single trade covering the next six weeks. Rather than opening the full phase plan with hundreds of tasks from all trades, they create a new workspace, open the import dialog, and filter by team and date range — pulling in only the concrete crew's tasks for the relevant period. The session stays focused, the workspace stays clean, and when the session is done the workspace can be discarded. The tasks themselves remain in the project database, unaffected.

The Project Control Center is a dedicated settings panel for project-level configuration in V4. It sits alongside the existing workspace Control Center and contains the settings that apply across the entire project — labels, teams, variance reasons, and calendars. Changes made in the Project Control Center are immediately available in every workspace that belongs to the project.

The problem it solves

The V3 Control Center was well-regarded — it covered workspace settings in a clear, consistent panel and users navigated it comfortably. Teams, labels, and variance reasons that were previously workspace-local now need to apply across the whole project. Keeping them in the workspace CC would have obscured that distinction — a change to a team in one workspace would affect all workspaces, with no visual indication that the scope had changed.

The solution was to create a separate Project Control Center with its own visual identity — using the project's name, icon, and color — to make the scope immediately clear. The workspace CC remains unchanged for workspace-specific settings. The Project CC sits alongside it for anything that belongs to the project.

Key concepts

Project Control Center — A dedicated settings panel for project-level configuration. Displays the project's visual identity and contains settings that apply across all workspaces in the project.
Project label — A label stored in the project database, available for assignment on any workspace element across all workspaces — including notes and shapes. Workspace label upgrade — The process of converting workspace-only labels to project labels, triggered via the "upgrade workspace labels" button. Runs a label-only sync to the project database.

How it works

Opening the Project Control Center

The Project Control Center opens as a separate panel alongside the workspace CC. Calendars

Calendars define which days are working days for the purpose of calculating task dates and durations. In V4, calendars are project-level objects stored in the central task database. Multiple calendars can exist within the same project, and each task can be assigned its own calendar — so different tasks in the same plan can follow different working day rules simultaneously.

The problem it solves

In V3, calendar settings were part of the plan configuration. A workspace had one set of default working days — defined in the Timeline Settings — and individual days on the plan timeline could be manually overridden to mark working days as holidays or non-working days as working. All tasks in the workspace shared these settings.

The only per-task calendar control was a toggle on each task: "skip weekends and holidays." When on — the default — a task respected the workspace's non-working days in its date calculations. When off, the task ignored all non-working days and ran across all seven days. There was no way to assign genuinely different working patterns to different tasks or teams within the same workspace — a concrete problem for projects involving trades with different working week structures, or teams operating across different public holiday calendars.

If "skip weekends and holidays" was on and a task started on a non-working day, a warning appeared on the task. Resolving that warning required either adjusting the task's start date or toggling the setting — there was no finer control available.

The other limitation was that the single workspace calendar also drove the plan visualization — off-days were shown as shaded columns on the plan itself, which worked when all tasks shared the same calendar but became misleading as soon as tasks needed different working day rules.

Key concepts

Calendar — A project-level object that defines working days and exceptions. Used to calculate task dates and durations and to determine how off-days are visualized on task shells.
Default working days — The day-of-week pattern for a calendar, defined as a set of enabled days (e.g. Monday through Friday). Applied to all dates unless overridden by an exception.
Calendar exception — A date-specific override within a calendar. A working day can be marked as a day off (e.g. a public holiday), or a non-working day can be marked as working (e.g. a scheduled Saturday shift).
Calendar hierarchy — The order in which calendars are resolved for a task: task calendar → team calendar → default calendar → project fallback calendar. The first level with a calendar defined wins.
Project fallback calendar — A Monday–Friday, five-day working week calendar that is always present in every project. It is not visible in the calendar list and cannot be edited. It applies to any task that has no calendar defined at any other level. If a project requires a different default pattern, a new calendar must be created and set as the project default.
Team calendar — A calendar assigned to a team in the Project Control Center. Applies to all tasks belonging to that team unless a task has its own calendar.

How it works

The calendar hierarchy

In V4, each task resolves its working day rules through a four-level hierarchy. The first level that has a calendar defined wins:

  1. Task calendar — a calendar assigned directly to an individual task. Overrides everything else
  2. Team calendar — a calendar assigned to the team the task belongs to. Applies to all tasks for that team unless a task has its own calendar
  3. Default calendar — a project-level calendar set as the default. Applies to any task or team without a more specific calendar
  4. Project fallback calendar — a Monday–Friday, five-day working week calendar that is always present in every project. It is not visible in the calendar list and cannot be edited. It applies when nothing else is set. If a project requires a different default pattern, a new calendar must be created and set as the project default

Creating and managing calendars

Calendars are created and managed in the Project Control Center. Each calendar has a name, a set of default working days defined by day of week, and a list of date-specific exceptions — days that override the default pattern in either direction. A working day can be marked as a day off, and a non-working day can be marked as a working day. Calendars created in the Control Center are available for assignment across the entire project.

Assigning calendars

A calendar can be assigned at three levels:

  • Project default — in the Project Control Center, setting a calendar as the default applies it to all tasks and teams that don't have a more specific assignment
  • Team — in the team settings within the Control Center, a calendar can be selected for a specific team. Changing a team's calendar updates the working day rules for all tasks assigned to that team that don't have their own task-level calendar override
  • Individual task — in the task editor on the General page, a calendar can be assigned directly to a task using the calendar picker

How off-days are visualized

Off-days are no longer shown as shaded columns on the plan. Instead, each task shell is responsible for visualizing its own non-working days — days within a task's duration that are non-working are shown with a hatched overlay directly on the shell. This is what makes it possible for tasks with different calendars to coexist in the same plan without visual conflicts. A full description of this visualization is covered in the Task Visualization guide.

Migrating calendars from V3

When a V3 workspace is synced to V4, calendar data is preserved and converted automatically. The sync creates one calendar per workspace based on that workspace's default working days and any day-specific exceptions from the plan timeline. A single seven-day calendar — with all days marked as working and no exceptions — is also created once per project and shared across all workspaces that need it.

How tasks are assigned during sync depends on their V3 setting:

  • Tasks with "skip weekends and holidays" set to on are assigned the workspace calendar, including any day-specific exceptions from the original plan timeline
  • Tasks with "skip weekends and holidays" set to off are assigned the seven-day calendar, so they continue running across all days without restriction

What's different from V3

  • Project-level calendars, not workspace-level — in V3, calendar settings were part of the plan configuration and applied to everything in the workspace. In V4, calendars are project-level objects that can be defined once and shared across all workspaces
  • Multiple calendars per project — different tasks and teams can follow different working day rules simultaneously within the same plan. This was not possible in V3
  • Calendar hierarchy replaces the per-task toggle — the "skip weekends and holidays" toggle on each task is replaced by the calendar hierarchy. Assigning a seven-day calendar to a task achieves the same effect as the old toggle-off, while team and project calendars provide structured control that the toggle never offered
  • Off-days visualized on the task shell, not the plan — in V3, non-working days were shown as shaded columns on the plan timeline, driven by the single workspace calendar. In V4, each task shell shows its own off-days based on its assigned calendar
  • Calendar data preserved on V3 migration — the workspace sync converts V3 calendar settings into proper V4 calendar objects and assigns them to tasks based on their original "skip weekends and holidays" setting

What this means for your team

Example 1 — Mixed working weeks in a single plan

A project has two active trades working simultaneously — a concrete crew on a six-day week including Saturdays, and a finishing trade on a standard five-day week. In V4, each team is assigned its own calendar in the Project Control Center. Both teams' tasks sit in the same plan, but date calculations and off-day visualisation are handled independently — the concrete crew's tasks don't show Saturdays as non-working days, while the finishing trade's tasks do. No workarounds, no toggling per task.

Example 2 — Switching to a six-day week for a single task

A trade is running behind on a task with a fixed end date that cannot move. Rather than replanning around it, the superintendent and foreman agree to work the coming Saturday to make up the time. A six-day calendar is assigned directly to that task in the task editor — overriding the team's default five-day calendar for that task only. The task's duration is recalculated accordingly and the Saturday shift is reflected in the plan. The rest of the team's tasks are unaffected.

Permissions

Project permissions control who can access a Hoylu project and what they can do within it. In V4, access is managed at the project level — not per workspace. Every user in a project has a project role that acts as a ceiling for what they can do in any workspace within that project. Workspace-level access can be adjusted within that ceiling, but never above it.

The problem it solves

In V3, managing permissions at the workspace level was a natural fit — data lived in workspaces, so controlling access per workspace controlled what users could affect.

V4 changes the underlying reality. Task data now lives at the project level. A task shell in one workspace is the same record as a shell in any other workspace — editing it anywhere affects it everywhere, including workspaces the editor may not explicitly have access to. Workspace-level permissions alone can no longer contain the scope of what a user can change.

There is also a practical UX problem. In a V4 workspace, live tasks and workspace-only elements like notes sit side by side in the same workspace. Without a consistent project-level permission model, the rules for what a user can edit become incoherent — why can you edit a note but not the task next to it? The answer V4 takes is to make project access the single controlling factor: if you have project write access, you can edit elements in a project workspace. If you are a project viewer, you cannot edit anything — live tasks or workspace-only elements. A workspace can still be locked down for presentation or reference purposes, but that is a deliberate workspace-level configuration applied on top of the project permission model, not a separate permission system.

This required building a clear, consistent project-level permission model — one where every user's access level is defined at the project and applied consistently across all workspaces within it.

Key concepts

Project role — The access level a user holds across the entire project. One of: Project Admin, Project Contributor, or Project Viewer. Acts as a ceiling for workspace access.
Project Admin — The highest project role. Can manage project membership, add and remove workspaces, and automatically has Admin access on every project workspace.
Project Contributor — A project member who can work within workspaces but cannot manage project membership. Workspace access can be set to Admin, Editor, or Viewer.
Project Viewer — A project member with read-only access. Can only be granted Viewer access on workspaces.
Workspace Admin — A workspace-level role that allows configuration of workspace settings, tools, and layout. User management has moved to the project level — Workspace Admins can no longer invite or remove users.
Public sharing — A workspace setting that controls whether non-project members can access a workspace via a direct link. When enabled, users who open the link are automatically added to the project with a chosen role.

How it works

Project roles

Every user in a project has one of three project roles. These roles act as a ceiling — workspace access can be set below this level but never above it.

Project Admin — can add and remove users and workspaces from the project, and automatically has Admin access on every workspace in the project.

Project Contributor — cannot manage project membership or workspaces, but can be granted Admin, Editor, or Viewer access on individual workspaces.

Project Viewer — cannot manage project membership or workspaces, and can only be granted Viewer access on individual workspaces. Editor or Admin workspace access is not available to Project Viewers.

Workspace access

Within a project, each workspace can have its own access level set for individual users — but always within the bounds of their project role. A Project Contributor can be given Admin, Editor, or Viewer access on a specific workspace. A Project Viewer can only ever have Viewer access, regardless of what workspace settings say.

Workspace Admins retain the same capabilities as before — configuring plan settings, showing and hiding tools, locking elements, and managing how a workspace is set up. What has changed is that Workspace Admins can no longer invite or remove users. User management is now exclusively a project-level action, and only a Project Admin can grant Workspace Admin access — and only to users who are at least Project Contributors.

The sharing dialog

The workspace sharing dialog has been updated to reflect the project-centric model. It displays the project name and color at the top, shows all project members alongside their project role, and supports sorting and searching to make it easier to find the right person in larger teams. Adding users is now handled entirely through the project rather than per workspace.

Public sharing

Each workspace has a Public Sharing & Default Access setting that controls whether non-project members can access the workspace via a direct link. When enabled, any user who opens the workspace via a link is automatically added to the project with a selected role — either Viewer or Contributor. When disabled, only users already explicitly added to the project can open the workspace, even with a direct link.

When a non-member opens a public sharing-enabled workspace, they are shown a prompt informing them that they will be added to the project and become visible to other project members. They can choose to continue or cancel.

What's different from V3

  • Access is now managed at the project level — in V3, permissions were primarily workspace-level. In V4, every user must be a project member with a project role before they can access any workspace in that project
  • Three project roles replace Admin and Basic — the previous Admin/Basic distinction has been replaced with Project Admin, Project Contributor, and Project Viewer. Each role has a concrete definition of what it allows across all workspaces
  • Project roles act as a ceiling — workspace access can be configured below a user's project role but never above it. A Project Viewer cannot be granted edit access on any workspace
  • Editing anything requires project write access — in V4, project access is the single controlling factor for editing. A Project Viewer cannot edit any element in a project workspace — live tasks or workspace-only elements like notes
  • Workspace Admins can no longer manage users — user management has moved entirely to the project level. Only a Project Admin can add or remove users and grant workspace access
  • The security gap is closed — in V3, sharing a workspace link could give non-project members access. In V4, non-project members cannot open any workspace even with a direct link unless public sharing is explicitly enabled
  • Public sharing replaces Link Sharing — the previous Link Sharing setting has been replaced by Public Sharing & Default Access, which includes a transparent opt-in prompt for new members
  • The sharing dialog shows project context — the redesigned dialog shows project name, color, and all project members alongside their roles, replacing the previous per-workspace user list

What this means for your team

These are examples of how teams use the new permission model in practice. Each one is independent — you can adopt any of them without the others.

Example 1 — Onboarding a team without manual invitations

A project admin sets up a new workspace at the start of a project and enables public sharing with the "non-members are added to the project" option set to Contributor. They share the workspace link in an email to all trade partners. As each person opens the link, they are prompted to confirm they are joining the project and are automatically added as Contributors — no manual invitation process, no back and forth. Once the full team is in, the admin disables public sharing. From that point, the project is closed and only explicitly added members can access it.

A GC project admin sets up three workspaces: a high-level milestone workspace, a phase planning workspace, and a weekly work plan workspace. The owner's representative is added to the project as a Project Viewer and given access only to the milestone workspace. They can open it at any time and see live project status — without being able to access the operational planning workspaces, edit anything, or get drawn into day-to-day detail. The GC retains full control over what the owner sees.

A programme manager prepares a milestone workspace for a monthly steering committee meeting. Before the session, they lock the workspace so that no participant can accidentally move or edit anything during the presentation. Project Viewers in the room can follow along on their own devices, while the PM walks through the live data. After the session, the lock is removed and the team resumes normal planning.

Public API & Third-Party Integrations

The Hoylu Public API gives external systems programmatic read and write access to project data in V4. It is built on the same central task database that powers workspaces, insights, and the mobile app — meaning data read or written through the API is always in sync with what users see in the product. Access is controlled through service tokens, which are created and managed within Hoylu and tied to a specific user's project permissions.

The problem it solves

Construction projects generate planning data that needs to flow in multiple directions. Schedulers work in dedicated planning tools. Project managers want live dashboards in reporting platforms. Site teams use specialist tools for safety, procurement, and logistics. In V3, there was no way for any of these systems to read or write Hoylu task data programmatically — the only way to get data in or out was through the product interface or manual exports.

The Public API changes this. It makes task, milestone, and constraint data available to any system that can make an HTTP request. A reporting tool can pull live task status to build a dashboard without anyone in Hoylu having to export anything. A scheduling tool can push updates back into the project without users having to re-enter data. Integrations that previously required manual synchronisation can now be automated.

Because the API reads from and writes to the central task database, every change made through the API is immediately visible in workspaces, in Insights, and in the mobile app — the same as any change made by a user.

Key concepts

Service token — A credential that grants an external system access to the Hoylu API on behalf of a specific user. Inherits that user's project permissions. Can be read-only or read-write. Valid for up to one year and can be revoked at any time.
Read token — A service token that allows an external system to query task, milestone, and constraint data. Cannot create or modify records.
Write token — A service token that allows an external system to create and update task, milestone, and constraint records. Requires the creating user to have Contributor or higher access.
Rate limit — A cap on the number of API requests that can be made within a given time window. Ensures consistent performance across all users and integrations.

How it works

Service tokens

Access to the API is granted through service tokens — long-lived credentials that can be created by any user with project access. A service token inherits the permissions of the user who created it: if that user has read-only access to a project, the token can only read from that project. If the user has write access, the token can also create and update records.

Tokens can be created, listed, and revoked through the token management interface. Revoking a token immediately removes its access — it cannot be undone. Each token is scoped to the creating user's permissions, which means access can be managed through the existing project permission model without any additional configuration.

Tokens have a maximum validity of one year. Organisation admins can view and revoke all tokens created within their organisation, giving IT teams oversight of third-party integrations.

Read access

A read token can retrieve tasks, milestones, and constraints from any project the creating user has access to. Data can be filtered by team, status, label, date range, and other fields — so a reporting integration can pull only the data it needs rather than downloading the full project. A filter by last-modified date is also available, allowing integrations to fetch only records that have changed since their last sync rather than re-downloading everything on every request.

Write access

A write token can create new tasks, milestones, and constraints, and update existing ones. This enables integrations where external planning or scheduling tools push data into Hoylu — for example, updating task dates or status after a schedule has been revised in an external tool. Write access requires the creating user to have Contributor or higher project access. All changes made through a write token are attributed to the creating user and appear in the task history like any other edit.

Rate limiting

API requests are rate-limited to ensure consistent performance for all users. The limits are set at levels that support regular integration workloads — polling for updates, pushing schedule changes, populating dashboards — without affecting the product experience.

What can be integrated

The API is the integration layer for any tool that works with project schedules and task data. Typical use cases include connecting Hoylu to BI and reporting platforms like Power BI, pushing data from master schedule tools, pulling live task status into safety or procurement workflows, and building custom dashboards or visualisations on top of project data.

What's different from V3

  • V3 had no public API — there was no way to read or write Hoylu task data programmatically in V3. All data access required using the product interface or manual export
  • Live data, no export required — because the API reads from the central task database, data is always current. There is no batch export or sync step needed to get an up-to-date picture of project data
  • Both read and write are supported — the API supports not just reading data out but pushing updates in, enabling two-way integrations with external planning and reporting tools
  • Permission-based access model — service tokens inherit the creating user's project permissions, meaning no separate permission model is needed for integrations. Revoking a user's project access automatically invalidates their tokens

What this means for your team

Example 1 — A live Power BI dashboard fed directly from Hoylu

A VDC director at a GC wants to give senior leadership a programme-level view of PPC trends and milestone health without them having to open Hoylu. They create a read token, connect it to Power BI, and build a dashboard that pulls task status, PPC data, and milestone dates directly from the project database. The dashboard updates automatically whenever data changes in Hoylu — no exports, no manual refreshes, no one having to prepare a report before the Monday morning meeting.

A customer builds a lightweight integration between their BIM coordination tool and Hoylu. When an issue is raised in the BIM platform — a clash, an RFI, an unresolved design question — the integration looks up the relevant Hoylu task by ID and uses the API to add a link to the BIM issue directly on that task. Foremen and planners opening the task in Hoylu can navigate straight to the issue in the BIM tool from the Connections page, without having to search for it separately. The connection between the planning record and the coordination issue is maintained automatically as long as the integration runs.

The Hoylu API makes it possible to connect project data to AI tools and assistants. A team could build an integration that allows an AI assistant to both read from and write to live project data through natural language. On the read side, the assistant could answer questions like "which tasks are overdue this week?" or "which trades have the most incomplete commitments?" or "draft a status update based on today's PPC" — all without opening Hoylu. On the write side, the same assistant could take action: a foreman arriving on site in the morning asks the assistant for all of their tasks for today, gets a spoken summary, and at the end of the shift simply tells it "mark all of today's tasks as complete" — the assistant updates the task status in Hoylu via the API. The plan stays current without the foreman ever opening the app.

XER Import & P6 Integration

🚧 This entire feature area is in active development and not yet available in V4. The description below reflects the intended design. Details may evolve before release.

XER Import V4 changes how P6 master schedules connect to Hoylu. The key shift is that the full XER file is imported into the project database first — every activity, calendar, WBS node, activity code, and UDF — and users then bring a relevant subset of those activities onto a workspace when they need them.

This two-step approach means the project retains the full context of the master schedule at all times. If a planner needs to add activities to a workspace later — because scope has expanded or they want to look at a different phase — they can do so directly from the project database without going back to the XER file. The same applies to fields: rather than having to select which activity codes or UDFs to import upfront, all fields are available in the project and users can choose which ones to display at any point.

V4 also addresses two other gaps in the current integration: activities and tasks competing visually on the same workspace, and no way to feed actuals back to P6. The existing subtask workflow is preserved as-is.

The problem it solves

The current XER import reads target start and end dates only. Because calendars are ignored, date calculations in Hoylu may not reflect what schedulers see in P6 — activities that span non-working days will appear with different durations depending on which side you look at them from.

Activity codes and UDFs present a different problem. They can be imported today, but users have to select which fields to bring in at import time. If they later want to work with a field they didn't include, the only option is to re-import the entire file. There is also no way to use WBS structure as a grouping dimension — the hierarchy exists in P6 but is not carried across to Hoylu.

On the workspace, imported P6 activities and Hoylu tasks are rendered at the same visual weight — same size, similar styling — which makes plans harder to read than they should be, particularly on dense programmes where the master schedule and the field plan need to stay visually distinct.

And when field teams complete work ahead of or behind schedule, there is no path from Hoylu back to P6. Actuals captured in Hoylu have to be manually re-entered into the master schedule by the scheduler, creating a gap between what happened on site and what the master schedule reflects.

V4 addresses all four of these gaps.

Key concepts

Activity — A Primavera P6 schedule element imported from an XER file. Stored in the project database as a distinct record type, separate from Hoylu tasks. Activities are excluded from PPC calculations — keeping the CPM and LPS worlds appropriately separate.
XER file — The native export format of Primavera P6. Contains activities, calendars, WBS structure, relationships, activity codes, UDFs, and resource assignments.
WBS (Work Breakdown Structure) — The hierarchical grouping structure of a P6 schedule. In V4, WBS becomes a first-class grouping dimension — plans can be organised by WBS level just as they can by team or label category today.
Activity Report — A dedicated view in Project Insights listing all imported activities, filterable by state (warnings, changes) and other criteria. Serves as the control point for reviewing and triggering write-back to P6.
Write-back — The process of exporting an updated XER file from Hoylu containing actuals (Actual Start, Actual Finish, Physical % Complete) ready to import into P6. Operates by updating existing activities only — new activities are never created from Hoylu.
Subtask — A Hoylu task created directly from an imported activity. Connected to the parent activity to keep field planning anchored to the master schedule without being rigidly constrained by it. If a subtask extends beyond the activity's end date, a visual conflict warning appears on both the subtask and the activity.

How it works

What's already working in V3

The current XER import solves several real problems for field planning teams. Team information can be flexibly mapped from resources, activity codes, or UDFs at import time. Imported activities are visually distinct from Hoylu tasks and excluded from PPC calculations.

The most valuable existing workflow is subtask creation. Field planners can create Hoylu tasks directly from an imported activity — the subtask is connected to the parent activity, and any subtask that extends beyond the activity's end date triggers a visual conflict warning on both elements. This keeps field planning anchored to the master schedule without being rigidly constrained by it.

Users can navigate between activities that have warnings, making it easy to jump from one flagged activity to the next. A basic CSV export of flagged activities also exists, though the UX for this is not yet straightforward.

Richer data in

V3: Target start and end dates only. Calendars and WBS ignored. Activity codes and UDFs require re-import to change.

V4: The full XER file is imported into the project database — every activity, calendar, WBS node, activity code, and UDF. Users then bring a relevant subset of those activities onto a workspace when they need them, and can add more at any point without going back to the XER file.

  • Calendars stored and respected — date calculations honor working days per activity, using the same calendar model as Hoylu tasks
  • WBS becomes a grouping dimension — create a plan view grouped by WBS level 3, or any level, just like grouping by team or label category today
  • All activity fields, UDFs and activity codes available — all fields are imported and stored; users decide which ones to display rather than selecting them at import time

A clearer visual language

V3: P6 activities and Hoylu tasks are rendered at the same visual weight — same size, similar styling — which makes plans harder to read on dense programmes.

V4: Activities look like activities. Tasks look like tasks.

  • Slim CPM-style bars — familiar to schedulers, clearly subordinate to Hoylu tasks visually
  • Dual bars — target (planned) dates shown as a background bar, actuals overlaid on top — classic tracking Gantt style
  • Distinct milestone visualization — milestones are displayed differently from activities to make them immediately recognizable
  • Critical path badge — toggle on a "CP" indicator on affected activities; no reliance on connection lines which are hidden by default
  • WBS relationships on hover — hover over any activity to see its siblings and parent/child connections light up

Activity Report

A dedicated report page in Project Insights — similar to the Constraint Log — giving schedulers a structured view of all imported activities without requiring them to navigate the workspace.

The report shows all activities with filters to narrow down the view. The most important filter is activity state:

  • Warnings — activities with a subtask conflict (a subtask extends beyond the activity's end date). This does not mean the activity has changed yet; it means a potential issue has been flagged that the scheduler needs to review.
  • Changes — activities where actual dates or progress have been updated in Hoylu, representing data that is ready to be written back to the master schedule.

The report also serves as the control point for the XER export — schedulers can review which activities have changes, decide which ones to include, and trigger the write-back from here rather than exporting everything at once.

Writing actuals back to P6

V3: No write-back exists. Actuals captured in Hoylu have to be manually re-entered into P6 by the scheduler.

V4: An updated XER file can be exported directly from Hoylu, ready to import into P6. During the export, the user provides an up-to-date XER file — Hoylu inserts the changes into it rather than generating a file from scratch. This ensures the write-back is always based on current P6 data and avoids overwriting anything that has changed in the master schedule since the last import.

Design decisions:

  • Update existing activities only — creating new activities in a P6 master schedule from an external tool is considered bad practice in the scheduling industry. P6 activity IDs, WBS placement, logic ties, and resource assignments are tightly controlled by the scheduler. Introducing new activities from outside risks corrupting the schedule's logic network. There is also no reliable way to round-trip new activities back into Hoylu after they are created in P6.
  • Fields written back: Actual Start, Actual Finish, Physical % Complete

Re-importing an updated schedule

When the scheduler sends an updated XER, users can re-import it with one action. Hoylu silently updates matched activities in place — no staging screen, no approval workflow. The task history handles the audit trail.

What's different from V3

  • Full XER stored in the project database — the entire file is imported upfront; users control what appears on a workspace, not what gets imported. Activities, fields, and structure can be added to a workspace at any point without re-importing from the XER.
  • All activity fields, activity codes and UDFs available without re-import — all fields are stored at import time and can be surfaced as labels or filters at any point afterward.
  • Calendars imported and applied — date calculations now respect the working day rules defined in the P6 schedule, rather than ignoring them.
  • WBS stored as a first-class grouping dimension — plan views can be organised by WBS level, not just by team or label category.
  • Distinct visual language for activities — slim CPM-style bars with dual target/actual display, clearly subordinate to Hoylu tasks on the workspace.
  • Activity Report in Insights — a structured, filterable view of all activities including warning and change states; serves as the write-back control point.
  • Write-back to P6 — no equivalent exists in V3. Actuals (Actual Start, Actual Finish, Physical % Complete) can be exported back to P6 via an updated XER file.

What this means for your team

Example 1 — A scheduler and a superintendent working from the same data

A scheduler maintains the master programme in P6 and exports an XER at the start of each phase. The superintendent imports it into Hoylu and creates subtasks directly from the P6 activities — one or more Hoylu tasks per activity representing the day-to-day field planning work. As subtasks are completed, the superintendent's plan updates in real time. At the end of the week, the scheduler opens the Activity Report in Insights, reviews which activities now have actuals, and exports an updated XER directly from Hoylu. The P6 schedule reflects what actually happened on site without anyone re-entering dates manually.

Example 2 — Reading a dense programme without losing the field plan

A project manager opens a workspace that shows both the P6 master schedule and the field-level tasks in the same plan. In V3, activities and tasks looked similar enough that the workspace became hard to navigate. In V4, the slim CPM-style bars sit clearly beneath the task shells — the hierarchy is immediately readable at a glance. The project manager can hover over any activity to see its WBS siblings light up, giving structural context without cluttering the workspace permanently.

Example 3 — Grouping by WBS instead of trade

A programme manager wants to see the plan organised by WBS level 2 rather than by trade. In V4, WBS is a grouping dimension just like team or label category — the plan can be rebuilt with one selection, showing a swimlane per WBS node. This is particularly useful on complex infrastructure projects where the master schedule's WBS structure maps more naturally to reporting than the trade breakdown used in day-to-day planning.

Frequently asked questions

Why does Hoylu only write back to existing activities — why not create new ones?
Creating new activities in a P6 master schedule from an external tool is considered bad practice in the scheduling industry. P6 activity IDs, WBS placement, logic ties, and resource assignments are tightly controlled by the scheduler. Introducing new activities from outside risks corrupting the schedule's logic network.
What happens when the scheduler updates the master schedule and re-imports?
Matched activities are updated in place silently — no staging screen or approval workflow. Changes are recorded in the task history. Activities that no longer exist in the updated XER are flagged rather than deleted automatically.
Is the subtask workflow changing in V4?
No. The ability to create Hoylu tasks directly from a P6 activity is preserved as-is. The conflict warning when a subtask extends beyond the activity's end date also remains unchanged.
What fields are written back to P6?
Actual Start, Actual Finish, and Physical % Complete. These are the fields most commonly used by schedulers to track progress in P6.
